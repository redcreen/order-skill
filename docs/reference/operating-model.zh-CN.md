# Order 运行模型

## 这份文档回答什么

这份文档回答：

把 `order` plugin 安装并绑定到指定 agent 后，agent 应该怎么工作。

它不讨论系统是什么，而讨论系统怎么跑。

## 安装后立刻生效的行为

plugin 完成安装并绑定 target agent 后，agent 必须立刻知道：

- 自己负责订单履约链路
- 所有输入先持久记录
- 与 order 无关的内容不进入正式业务线程
- 自然语言输入不能直接写正式业务表
- 所有正式录入都必须走草稿、确认、落库

## 安装范围与硬执行

`order` 不应该默认全局安装到所有 agent。

要求是：

1. plugin 必须显式绑定到一个 target agent
2. 未绑定前，实际执行入口不可用
3. order 相关动作必须调用真实脚本
4. 只有脚本成功，才能回报“已记录”“已落库”“已刷新日报”

## 输入处理模型

## 第一步：全部输入先记下来

所有用户输入都要先持久化。

原因：

- 防止崩溃和重启丢失
- 防止 `/new` 后现场信息消失
- 支持补录和关系后补
- 支持后续对象线程连续性

注意：

“全部输入都要记录”不等于“全部输入都进入正式业务事实”。

## 第二步：判断是否进入 order 业务线程

系统先判断这条输入是否与 order 有关。

### 进入 order 线程

如果涉及：

- 样品
- 报价
- 订单
- 生产
- 发货
- 回货
- 应收应付
- 收付款
- 开票
- 入库
- 库存
- 退货 / 修理 / 返工

则进入 order 业务线程。

### 不进入 order 线程

如果只是：

- 闲聊
- 泛讨论
- 非 order 话题

则只保留原始输入层，不进入正式业务线程。

## 第三步：自然语言引导

用户不会按字段模板说话，这是默认前提。

所以系统必须：

1. 接住自然语言
2. 抽取已知字段
3. 判断关键缺口
4. 只追问当前最关键的 1 到 2 个问题
5. 形成草稿

### 错误方式

一上来给用户一个标准录入表单。

### 正确方式

先复述当前理解，再追问下一个关键缺口。

例如：

> 我先记到草稿里了。  
> 目前识别到：客户林超，大红马 30 个。  
> 还差规格和客户确认单价，补完后我再正式建单。

## 草稿、确认、落库

正式写入前，系统必须始终经过：

1. `draft`
2. `confirmation`
3. `commit`

### 不允许直接落库的情况

- 关键字段缺失
- 对象关系不明确
- 金额 / 数量不确定
- 高风险修改未确认

### 必须明确确认的动作

- 新建正式订单
- 修改金额
- 修改数量
- 记录收款 / 付款
- 记录开票
- 关闭订单
- 删除 / 作废 / 冲销

## 乱序与补录处理

这在真实业务里是常态，不是异常。

### 系统处理原则

1. 先把事实接住
2. 先形成候选对象
3. 暂时不清楚归属时进入待关联池
4. 后续再补关系
5. 最终再确认正式归属

### 典型场景

- 先来了付款截图，后面才知道对应哪张单
- 一笔付款对应多个对象
- 先发货，后补物流号
- 先收到工厂反馈，后补批次归属

## 流程模板驱动

不同产品和不同批次流程可能不一样。

真实用户通常不会这样输入：

> 实际环节是 采购布料 -> 复合 -> 激光下料 -> 补裁片 -> 车缝充棉手工全流程 -> 质检

更常见的输入会是：

> 小兔子这批客户确认了 500 个，按上次那个做法走，布料到了，刘旭那边还要不要切你帮我看下。

因此流程不能要求用户标准化填写，而必须由系统主动带出来。

系统运行时必须支持：

- 先查该产品有没有默认流程模板
- 如果有，系统先给出“建议流程”，让用户确认、删除、插入或调整工序
- 如果没有或不沿用，系统一步步引导确认这批货的实际流程
- 每个产品可以有自己的默认流程；同一产品不同规格也可以有不同流程
- 每一批订单都要形成自己的“本批次流程计划”，不能只引用模板
- 本批次流程计划可以随时被修订，但修订必须保留版本、原因、来源输入和确认记录
- 对影响交期、成本、应付账款或库存动作的流程变更，必须重新确认后才能正式生效

后续的：

- 录入
- 跟单
- 提醒
- 日报

都要基于“本批次实际流程计划”来运行。

### 产品流程确认对话

当用户只输入产品、数量、客户或某个现场动作时，系统的正确行为是：

1. 识别产品和规格。
2. 查找该产品/规格的默认流程模板和最近几批实际流程。
3. 给出建议流程摘要，而不是要求用户自己写流程。
4. 明确标出不确定项，例如是否复合、是否刺绣、是否定位切、是否分车缝/充棉/手工。
5. 让用户用自然语言确认或修复。
6. 用户确认前，只能形成 draft / checkpoint，不能创建正式 work orders。

示例：

```text
我理解这批“小兔子 18cm 粉色”默认流程是：
采购布料 -> 激光下料 -> 定远乡冯杰车缝+充棉+手工 -> 质检。

这次是否沿用？
我还需要确认：
1. 是否需要复合？
2. 是否需要刺绣？
3. 是否有补裁片或补配件？
4. 是否直接发客户，还是回义乌云仓？
```

用户可以这样修复：

```text
这批不用刺绣，要先复合再切，车缝还是冯杰，充棉手工也在定远乡做。
```

系统应把这句话转成新的本批次流程计划，并再次给出确认摘要。

### 流程模板和本批次计划的关系

- `process_templates` 是产品可复用经验，不是本批次事实。
- `lot_process_plans` 是本批次确认后的执行计划。
- `work_orders` 必须从已确认的 `lot_process_plans` 派生。
- 如果流程只是系统建议，状态应是 `suggested` 或 `draft`。
- 如果用户确认，状态才变成 `confirmed`。
- 如果后续修改，必须生成新版本或修订记录，不能静默覆盖旧计划。

## 查询运行方式

用户不会查没关系，系统要会引导。

## 查询基本流程

1. 识别模糊查询意图
2. 先给一个合理默认范围
3. 如有必要，只补一个关键问题
4. 输出结果

### 示例

用户说：

> 王总那批现在做到哪了

系统应该：

- 判断是在查订单/批次当前进度
- 找到相关对象线程
- 必要时问“你说的是狗那批还是白兔子那批”
- 然后给出最终状态

## 主动跟单

系统必须内建跟单能力。

不能等用户想起来再查。

## 跟单的对象

系统要盯的不是“静态状态”，而是：

- 承诺交期
- 承诺发货时间
- 承诺回货时间
- 承诺付款时间
- 承诺确认时间

## 跟单的输出

系统应自动生成：

- `followup_items`
- `alerts`
- `exception_cases`

## 逾期处理

当承诺超时未完成，系统应：

1. 标风险
2. 生成跟进任务
3. 给出下一步建议
4. 必要时生成沟通草稿

## 日报

日报是运行模型里的正式能力，不是可选附加项。

日报应每天生成，并保持：

- 简洁
- 清晰
- 有具体行动建议

### 日报标准结构

1. 今日概览
2. 关键变化
3. 风险与阻塞
4. 资金视图
5. 明日行动建议

## 对外发送

系统对外通知应分三档：

### 1. `draft_only`

只生成草稿。

### 2. `confirm_then_send`

你确认后自动发送。

### 3. `policy_auto_send`

只有在明确授权的低风险固定场景下，才允许自动发送。

例如未来可能适合：

- 发货后自动发物流通知
- 入库后自动发内部通知

## `/new`、重启、会话切换后的恢复

系统不能依赖当前对话上下文继续工作。

恢复时应优先恢复：

- 当前相关对象线程
- 未完成草稿
- 待确认事项
- 跟进项
- 异常项

而不是简单恢复整段历史聊天。

## 记录策略

### 必须正式记录

- 影响履约和结算的已确认事实
- 证据附件
- 确认后的金额、数量、时间、对象关系

### 先入草稿

- 不完整
- 低置信度
- 关系不明确
- 需要多对多分配的事实

### 只保留原始输入

- 非 order 内容
- 泛闲聊
- 不形成业务动作的讨论

### 只保留审计

- 放弃的草稿
- 失败的写入
- 被拒绝的高风险动作
