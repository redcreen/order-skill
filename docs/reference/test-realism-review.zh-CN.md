# Order 测试真实性审查

## 结论

当前 `scripts/test_order_llm_chaos_stress_50.py` 的 50 案例测试不完全符合真实人类使用场景。

它已经验证了：

- 自然语言正文不再直接写 `实际环节是` 或 `代码顺序口径`
- LLM 能根据 `product_flow_templates` 输出期望流程
- 输出字段能 exact match
- runtime 能按期望流程顺序落库并保持数据一致性

但它还没有充分验证：

- 用户只输入一句很短的话
- 用户缺省大部分上下文
- 系统主动追问缺什么
- 系统给出候选流程让用户确认或修复
- 用户分多轮补充信息
- 未确认前只停留在 draft / checkpoint
- 确认后才生成正式订单、流程计划和 work orders

因此，这个测试应保留为“长文本抽取和 runtime 一致性测试”，但不能代表真实交互验收。真实交互验收改由 `scripts/test_order_lazy_guided_intake_50.py` 覆盖：它用多轮短输入验证 draft、checkpoint、流程确认和确认后落库。

## 当前测试输入画像

本轮审查基于 50 个自动生成案例统计：

| 指标 | 当前值 |
| --- | --- |
| 案例数 | 50 |
| 单条输入字符数 | 最少 1525，平均 1665，最多 1778 |
| 单条输入行数 | 固定 19 行 |
| 单条输入散消息数 | 固定 10 条 |
| 包含客户/产品/数量/单价/总金额 | 50/50 |
| 包含完整厂商清单 | 50/50 |
| 包含物流/异常块 | 50/50 |
| 包含售后/退款/扣款/已付供应商 | 50/50 |
| 包含内部场景标签 | 50/50 |
| 包含测试编号 | 50/50 |

这不是典型真实输入。真实用户更可能输入：

```text
王总小兔子 500 个，按上次做，定金到了 1800，冯杰那边先排。
```

或者：

```text
刘旭说小兔子裁片不够，要补一批。
```

或者：

```text
白熊那批客户说少了 20 个，先补发。
```

## 主要问题

### 1. 输入太完整

当前测试每条都主动给了客户、产品、规格、数量、单价、总金额、定金、订单类型、全部厂商、仓库、异常、退款、扣款、已付供应商。

这更像系统整理后的“现场摘要”，不像用户原始输入。

真实输入通常只有 1 到 2 个事实，其他事实需要系统从历史、产品模板、上下文、候选订单中找，或者主动追问。

### 2. 没有测试追问能力

当前测试要求 LLM 一次性输出完整 JSON。

真实系统应该经常输出：

```text
我先不正式录入，还缺这些确认：
1. 这个“小兔子”是王总 4 月那张单，还是新单？
2. 按上次流程走是否包含复合？
3. 冯杰只做车缝，还是车缝+充棉+手工都做？
4. 这笔 1800 是定金还是尾款？
```

当前测试没有验证这些 checkpoint 是否生成，也没有验证未确认时是否阻止正式落库。

### 3. 产品流程模板仍然以 prompt 方式直接提供

当前测试通过 `product_flow_templates` 把每个产品的默认流程放进 prompt。

这能验证“给了模板后能不能抽取”，但还没有验证：

- 系统能否根据产品名查本地模板
- 无模板时能否从历史订单归纳候选流程
- 多个相似产品模板时能否提出候选并让用户确认
- 用户修改模板后是否能保存为新的产品默认流程或本批次流程

### 4. 没有测试多轮输入

真实场景通常是：

1. 用户先说一句短消息
2. 系统提出草稿和缺失问题
3. 用户补一句
4. 系统更新草稿
5. 用户确认
6. 系统才正式落库

当前测试是单轮批量抽取，不覆盖这个核心体验。

### 5. 没有测试“不要记录无关内容”

真实 channel 里会混入大量和 order 无关的信息。

当前测试全是 order 相关内容，没有验证：

- 系统如何判断是否 order 相关
- 不相关内容是否只作为临时会话
- 模糊相关内容是否进入待确认队列

## 应新增的测试分层

### Tier 1: 长文本抽取一致性测试

保留当前 50 案例。

目标：

- 验证 `gpt-5.5` 路由
- 验证长文本抽取
- 验证 runtime 数据一致性
- 验证 work_orders 顺序和数量

这不是最终真实交互验收。

### Tier 2: 懒人短输入引导测试

新增 50 个短输入案例。

输入长度建议：

- 60% 案例少于 30 字
- 30% 案例 30 到 80 字
- 10% 案例 80 到 160 字

每条只包含部分事实，例如：

```text
小兔子王总 500 个，按上次做，定金到了。
```

预期结果不是直接落库，而是：

- 生成 draft
- 给出候选订单/产品/流程
- 生成缺失字段 checkpoint
- 要求用户确认

### Tier 3: 多轮确认测试

每个案例包含 3 到 6 轮输入。

示例：

```text
用户：小兔子王总 500 个，按上次做。
系统：我找到两个小兔子模板，是 18cm 粉色还是 20cm 白色？
用户：18 粉色，冯杰做，定金 1800 到了。
系统：是否沿用 采购布料 -> 激光下料 -> 冯杰车缝充棉手工 -> 质检？
用户：这批先复合再切。
系统：已更新为 采购布料 -> 复合 -> 激光下料 -> 冯杰车缝充棉手工 -> 质检，确认正式建单吗？
用户：确认。
```

预期：

- 前几轮不落正式业务表
- 确认后才落 `sales_orders`、`lot_process_plans`、`work_orders`
- 所有原始输入都持久化

### Tier 4: 乱序补录和关联测试

测试先出现付款、物流、账单、售后，再出现订单的情况。

预期：

- 先创建 pending association
- 后续识别归属后显式 resolve
- 一笔付款可分配到多个订单
- 一个供应商账单可关联多个产品或多个作业

### Tier 5: 无关内容过滤测试

混入和 order 无关的消息。

预期：

- 明确无关内容不进入正式 order 记录
- 模糊相关内容进入待确认
- 不因为关键词误判而乱建草稿

## 已新增测试层

已新增 runtime 引导脚本：

```text
scripts/test_order_lazy_guided_intake_50.py
```

这个脚本不复用当前“一次性抽取完整 JSON”的口径，而是验证：

- 短输入
- 缺字段
- 系统追问
- 草稿状态
- checkpoint
- 用户确认
- 确认后落库

已新增端到端 GPT-5.5 抽取脚本：

```text
scripts/test_order_llm_lazy_guided_intake_50.py
```

这个脚本把 GPT-5.5 短句抽取接到懒人引导链路前面，验证：

- OpenClaw order agent 实际使用 `openai-codex/gpt-5.5`
- `fallback_used=false`
- 模型从每轮短输入抽取字段增量
- 系统派生字段仍由 runtime 生成，而不是让 LLM 伪造
- 模型抽取最终必需字段 exact match
- runtime 继续执行未确认阻断、流程确认、正式落库和工单顺序校验

已新增订单后续混乱事件脚本：

```text
scripts/test_order_messy_event_confirmation_50.py
```

这个脚本不再只测“建单”，而是测订单存在后的真实零散事件：

- 客户付款但订单归属不清
- 应收补录
- 供应商账单
- 供应商付款
- 裁片发工厂物流
- 给客户发货
- 退货返修
- 退款挂账
- 供应商质量扣款
- 补裁片 / 返工工单
- 无关闲聊过滤

测试原则：所有正式业务写入都必须先进入 draft，未生成确认预览、假确认 token、待关联未 resolved，都不能写入正式表。

## 懒人输入测试结果

最近一次验证结果：

| 批次 | 结果 | 输入画像 | 数据结果 |
| --- | --- | --- | --- |
| runtime 10 案例 | 通过 | 30 条输入，平均 24.37 字，最长 35 字 | 10 个订单、59 个工单、0 个未关闭草稿 |
| runtime 50 案例 | 通过 | 150 条输入，平均 24.95 字，最长 36 字 | 50 个订单、295 个工单、0 个未关闭草稿 |
| GPT-5.5 + runtime 10 案例 | 通过 | 30 条输入，平均 24.37 字，最长 35 字 | 10 个订单、59 个工单、0 个未关闭草稿 |
| GPT-5.5 + runtime 50 案例 | 通过 | 150 条输入，平均 24.95 字，最长 36 字 | 50 个订单、295 个工单、0 个未关闭草稿 |
| GPT-5.5 + 混乱事件 50 案例 | 通过 | 50 条输入，平均 28.74 字，最长 37 字 | 46 个正式事件确认写入、4 个无关输入忽略、2 个结算分配确认写入、0 个未关闭草稿 |

50 案例覆盖：

- 150 条短句由 GPT-5.5 抽取
- 10 个模型批次全部使用 `openai-codex/gpt-5.5`
- 10 个模型批次全部 `fallback_used=false`
- 50 个案例最终必需字段 exact match
- 50 次首次输入缺字段阻断
- 50 次流程确认必需
- 50 次确认后提交
- 295 个 work order 顺序校验

混乱事件 50 案例覆盖：

- 50 条短事件由 GPT-5.5 抽取，并由系统标准化为 runtime 字段
- 10 个模型批次全部使用 `openai-codex/gpt-5.5`
- 10 个模型批次全部 `fallback_used=false`
- 46 个正式事件全部确认后写入
- 4 个无关闲聊不进入 order draft
- 5 个待关联付款先阻断，再人工 resolve 后提交
- 46 次假确认 token 被阻断
- `allocate` 直接调用无 token 被阻断
- `allocate` dry-run 不写入，只生成确认 token
- `allocate` 假 token 被阻断
- 2 个结算分配带确认 token 后写入
- 0 个未关闭 draft
- 0 个未解决 pending association
- SQLite integrity check 为 `ok`

注意：每批 10 个案例时曾出现模型输出 JSON 不可解析。端到端短句测试默认 batch size 改为 5，降低单次输出过长导致格式漂移的风险。

已解决风险：`allocate` 结算分配命令不再允许直接正式写入。它必须先 `dry_run=true` 生成确认 token；无 token、假 token、payload 被修改都会失败；只有带匹配 token 才会写入 `settlement_allocations` 并更新应收/应付/退款状态。

当前 `test_order_llm_chaos_stress_50.py` 可以继续保留，但命名或报告里应明确它是：

```text
LLM long-context extraction + runtime consistency stress test
```

而不是：

```text
真实用户交互验收测试
```
