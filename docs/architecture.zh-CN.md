# Order 架构

## 这份文档回答什么

这份文档回答一个稳定问题：

`order` 系统从全局上到底是什么，边界在哪里，为什么不能继续沿用旧的表格型和会话型设计。

## 系统目标

`order` 的目标不是做一个“更会聊天的录单工具”，而是做一套面向真实生产加工场景的订单运营系统。

这套系统应该能够：

- 接住零散、自然语言、乱序输入
- 在重启、`/new`、上下文收缩后保持连续性
- 把输入逐步收敛成结构化业务事实
- 主动发现没人跟的事情
- 给出日报和行动建议
- 为未来 ERP / 仓库系统桥接保留稳定边界

## 边界

### 系统内

`order` 范围内的对象：

- 样品
- 报价
- 订单
- 生产批次
- 工艺流程与作业
- 发货 / 回货
- 应收 / 应付
- 收付款
- 开票
- 入库 / 库存同步准备
- 跟单、提醒、日报、异常

### 系统外

当前不属于 `order` V1 的对象：

- 泛财务分析
- 总账
- 预算和税务
- 完整 ERP 主数据系统
- 完整仓库执行系统

这些内容以后如果要接，也只能通过明确的桥接边界接入，不能反向成为 order 的真相源。

## 三层记录模型

这是整个系统最核心的结构。

### 1. 原始输入层

所有用户输入都先持久记录下来。

目标：

- 防止系统崩溃、重启、`/new` 后数据丢失
- 支持后续补录和关系后补
- 支持跨会话连续性

### 2. 候选理解层

系统对输入做：

- 意图识别
- 字段抽取
- 候选对象关联
- 缺口判断

这一层允许不完整和低置信度存在。

### 3. 正式业务事实层

只有经过确认的事实，才能进入：

- 订单
- 批次
- 发货
- 应收应付
- 流水
- 发票
- 入库
- 库存变动

## AI 的职责

AI 的职责不是代替真相源，而是做“理解与串联层”。

它最适合负责：

- 理解自然语言
- 抽取候选字段
- 找候选关系
- 追问缺口
- 生成确认摘要
- 发现风险
- 生成日报和沟通草稿

正式真相源仍然应该是本地结构化存储。

## 真相源

### 正式真相源

- 本地 `SQLite`
- 本地文件系统附件归档

### 非真相源

以下内容都不能作为最终真相源：

- 会话上下文本身
- 历史聊天窗口
- 外部表格系统
- 外部 ERP 当前页面状态
- 模型当下的临时理解

## OpenClaw 注入模型

`order` 应该保持在 wrapper 路径上，而不是走 ingress 短路拦截。

当前宿主侧契约是：

- `before_prompt_build` 往真实会话里注入 order wrapper 契约
- 正常 agent flow 继续沿用同一个 transcript
- `before_tool_call` 会阻断直接执行 `order/scripts/order_runtime_api.py` 或其它 bundled `order/scripts/*.py`
- 真正的 runtime 动作必须通过 `plugins/openclaw-order/scripts/order_hard_execute.py`
- wrapper 只调用 `order/scripts/order_runtime_api.py`，OpenClaw 不持有业务路由逻辑

这样既保住了真实会话连续性，也防止模型绕过硬执行 wrapper。

## 对象线程连续性

这不是普通聊天系统，所以连续性不能只依赖“上一轮对话”。

系统要维护对象线程：

- 订单线程
- 样品线程
- 批次线程
- 发货线程
- 退货 / 返修线程
- 对方单位线程

这样在系统重启、`/new`、跨 channel 切换时，仍然能够恢复与当前对象相关的工作现场。

## 工作图而不是线性流程

真实业务不是一条直线。

它会出现：

- 加单
- 减单
- 取消
- 补采
- 返工
- 返修
- 退货
- 分批回货
- 分批交付

所以系统不能只用一个“订单状态”字段理解全部业务。

系统必须同时维护：

- 业务对象
- 业务事件
- 具体作业
- 作业之间的依赖和回流关系

## 流程模板是一级能力

不同产品、不同批次的加工路径可能不同。

因此系统必须支持：

- 产品默认流程模板
- 批次实际流程计划

后续的录入、跟单、提醒、日报都要围绕“这一批货确认过的流程”展开，而不是围绕一条想象中的通用流程展开。

## 主动跟单是系统内置能力

`order` 系统不应只记录“现状”，还要记录：

- 之前承诺了什么
- 什么时候该完成
- 如果没完成，下一步该谁跟

因此系统必须内建：

- commitments
- follow-up items
- exception cases
- alerts

## 对外通知必须受控

系统未来可以支持：

- 草稿生成
- 确认后发送
- 在明确授权条件下自动发送

但自动发送必须始终是“策略控制下的能力”，不能成为无边界行为。

## 对外系统桥接边界

未来如果接：

- 聚水潭
- 仓库系统
- 库存同步
- 入库联动

也必须通过桥接层接入。

order 系统负责运营真相，外部系统负责执行和回传，不能反过来把外部系统当成内部模型所有者。

## 非目标

当前 order 设计明确拒绝：

- 强迫用户按标准表单说话
- 把所有输入都直接写入正式业务表
- 让 AI 直接替代数据库

## 文档归属

- `docs/*` 负责 order-skill 专属 durable 文档
- `plugins/openclaw-order/*` 负责 plugin 分发说明
- `order/*` 负责 runtime 主体和运行时说明
